# UI 디자인 방향 연구 v0.1

> 목적: Spindle이 범용 여행 앱이나 AI 생성 SaaS 템플릿처럼 보이지 않도록 제품 고유의 시각·상호작용 원칙을 정한다.
> 실제 화면 구조와 카피의 구현 기준은 `docs/ui.md`를 따른다. 이 문서는 그 판단 근거와 디자인 언어를 정의한다.

## 1. 결론

Spindle의 디자인 기준점은 **"손에 든 항해 계기판이 여행 게임이 된 모습"**이다.

- 여행 정보를 탐색하는 목록형 앱이 아니라, 휴대폰을 돌려 방향을 정하는 단일 목적 도구로 보여야 한다.
- 개성은 장식의 양이 아니라 원·부채꼴·항로선·절취선처럼 제품 기능에서 나온 형태와 동작의 일관성으로 만든다.
- 화면에서 로고와 문구를 가려도 "방향을 돌려 장소를 발견하는 서비스"로 식별되어야 한다.

## 2. AI 생성 UI처럼 보이는 원인

AI 생성 UI의 문제는 완성도 부족보다 안전한 관습의 반복에 가깝다. 2026년 공개된 92명 대상 연구에서도 생성형 AI 프로토타입은 실용성 평가가 양호했지만 독창성과 혁신성 평가는 중립 또는 부정적이었다.

Spindle에서 피할 패턴:

- 파란색·보라색 그라디언트 위에 흰색 CTA를 놓은 히어로 배너
- 검색, 프로필, 빠른 메뉴, 추천 캐러셀, 5탭 내비게이션을 한 홈에 모은 플랫폼형 구조
- 모든 정보를 둥근 카드, 필, 상태 배지로 감싸는 구성
- 화면 전체에 동일한 라운드 값과 부드러운 그림자를 적용하는 방식
- 기능과 무관한 글로우, 유리 효과, 떠다니는 원형 장식, 반복 바운스
- 모델 시트 없이 생성한 글로시 3D 캐릭터와 포즈 변형
- 실제 데이터보다 기능 설명 문장이 더 많이 보이는 화면

완성된 디자인은 요소를 더 많이 넣은 화면이 아니라, 제품과 무관한 요소를 설명 없이 제거할 수 있는 화면이다.

## 3. 고유한 디자인 언어

### 3.1 핵심 형태

| 형태 | 의미 | 적용 |
|---|---|---|
| 원 | 나침반, 회전, 탐색 범위 | 홈 원판, 스핀 진행 |
| 부채꼴 | 선택된 방위 | 정지 연출, 방위별 색 |
| 항로선 | 이동과 연결 | 로딩, 화면 전환, 거리 정보 |
| 눈금·십자선 | 기준점과 정밀함 | 원판, 이미지 폴백 |
| 절취선·잘린 모서리 | 여행 표와 기록 | 결과 정보부, 공유 카드 |

- 일반 컨테이너의 모서리는 8~12px 범위로 제한한다.
- 원형과 큰 라운드는 나침반, 도장, 실제 조작부처럼 의미가 있을 때만 사용한다.
- 정보 구획은 그림자보다 1px 선, 여백, 배경 명도 차로 나눈다.
- 금속 나사나 황동 질감을 흉내 내는 사실적 스큐어모피즘은 사용하지 않는다. 항해 모티프는 재질 모사가 아니라 기하와 움직임으로 표현한다.

### 3.2 색

현재 `docs/ui.md`의 딥 네이비와 선셋 오렌지를 유지하되, 코발트 블루 중심의 SaaS 인상을 피한다.

- 밤바다: `#0F2540`
- 종이·해무: `#F3EEE3`
- 신호 오렌지: `#FF7A45`
- 항만 청록: `#287284`
- 먹색: `#182126`

8방위 보조색은 선명한 무지개가 아니라 바다, 해무, 노을, 콘크리트, 항만 표지에서 가져온 저채도 색으로 구성한다. 색만으로 방위를 구분하지 않고 방위명과 부채꼴 위치를 항상 함께 표시한다.

### 3.3 글자

- 본문과 조작부: Pretendard
- 방위명과 짧은 큐레이션 문장: 세리프 계열을 제한적으로 검토
- 제목을 모두 굵은 고딕으로 처리하지 않는다. 크기, 자간, 여백으로 위계를 먼저 만든다.
- 이모지는 시스템마다 모양이 달라지므로 핵심 UI 아이콘으로 사용하지 않는다.

### 3.4 이미지와 캐릭터

- 결과 화면의 주인공은 실제 장소 사진이다.
- 이미지가 없을 때는 방위 부채꼴, 항로선, 나침반 눈금으로 만든 단색 벡터 폴백을 사용한다.
- MVP에서는 글로시 3D 마스코트를 핵심 화면에서 제외한다.
- 마스코트를 다시 도입하려면 비례, 실루엣, 표정, 색상 수, 선 굵기, 금지 동작이 고정된 2D 모델 시트를 먼저 만든다.

## 4. 화면 구조

### 홈 / 스핀

- 첫 화면 면적의 대부분을 나침반 원판에 사용한다.
- 검색, 프로필, 추천 캐러셀, 범용 빠른 메뉴를 두지 않는다.
- 상단에는 모드와 출발점만 작게 표시한다.
- 이동시간은 원판 아래의 짧은 3단 선택기로 둔다.
- 핵심 행동은 현장 모드의 "내 위치에서 돌리기" 또는 여행 모드의 "돌리기" 하나다.

### 방위 공개

- 캐릭터와 보상 상자를 보여주는 대신 선택된 방위 부채꼴이 원판에서 직접 확장된다.
- 방위명과 한 줄 메시지가 같은 축에 나타나며, 탭하면 즉시 결과로 넘어간다.
- 기능과 무관한 별, 글로우, 반복 부유 애니메이션은 사용하지 않는다.

### 결과

- 상단은 장소 사진을 넓게 사용하고, 하단 정보 영역은 한 장의 여행 표처럼 구성한다.
- 배지·상태 아이콘·색상 칩은 사용하지 않는다. 방문 정보는 구분선이 있는 라벨–값 목록과 원문 텍스트로 전달한다.
- 주 행동은 `길찾기`, 보조 행동은 `다른 후보`, `다시 돌리기`, `공유 카드` 순으로 위계를 둔다.
- 알고리즘 설명은 별도 카드로 늘리지 않고 `왜 이곳인가요?` 한 줄 펼침 정보로 제공한다.

### 공유 카드

- 앱 화면 캡처처럼 만들지 않고 독립된 여행 포스터로 보이게 한다.
- 사진, 방위 부채꼴, 관광지명, 한 줄 메시지만 남긴다.
- 홈, 결과, 공유 카드가 같은 눈금·색·절취선 문법을 사용해야 한다.

## 5. 스핀 입력과 모션

### 현장 모드

- 스핀의 입력은 손가락 드래그 속도가 아니라 **휴대폰을 실제로 돌리는 속도**다.
- `DeviceOrientation`의 연속 방위각에서 0/360도 경계를 보정한 최단 각도 차이를 구하고, 시간 변화량으로 기기의 회전 속도를 판단한다.
- 화면 원판은 기기 방위각을 따라 움직인다. 휴대폰을 빠르게 돌리면 화면도 빠르게 돌고, 천천히 돌리면 천천히 움직인다.
- 사용자가 회전을 멈추고 방위값이 안정되면 그 시점의 실제 방위각을 잠그고 8방위로 분류한다.
- 화면만 추가로 도는 가짜 관성 때문에 최종 연출 각도와 실제 방위각이 달라져서는 안 된다.

### 여행 모드

- 여행 모드는 센서 없이 완전히 동작해야 한다.
- `돌리기` 버튼을 누르면 자동 스핀 애니메이션이 실행되고 하나의 방위에 정지한다.
- 버튼을 누른 시간이나 손가락 속도는 결과에 영향을 주지 않는다.

### 공통 모션 원칙

- 모션은 입력, 상태 변화, 결과 공개를 설명할 때만 사용한다.
- 스핀 연출은 언제든 탭으로 건너뛸 수 있어야 한다.
- `prefers-reduced-motion`에서는 연속 회전 대신 짧은 단계 전환 또는 페이드로 방위를 확정한다.
- 중요한 정보는 움직임만으로 전달하지 않는다.

## 6. 구현 전 삭제 우선순위

1. 홈의 검색·프로필·추천 캐러셀
2. MVP에 필요하지 않은 5탭 하단 내비게이션
3. 글로시 3D 마스코트와 글로우·별 효과
4. 반복되는 파란 그라디언트와 큰 그림자
5. 과도한 필·배지·둥근 카드
6. 사용자가 이미 보고 있는 상태를 다시 설명하는 문장
