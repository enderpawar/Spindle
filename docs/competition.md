## 핵심 규정 (FAQ 확인 사항)

1. **TourAPI 실시간 호출만 인정.** 로컬 DB 적재는 별도 신청서 필요, 파일 데이터는 불인정 → Spindle은 실시간 호출로 설계 (신청서 경로 사용 안 함).
2. **운영계정 인증키 기준으로 개발 기간 중 실제 호출 이력을 검증** → 개발 첫 주부터 운영계정 키 사용.
3. 생성형 AI 사용에 따른 감점 없음.
4. 웹 서비스는 URL 제출로 심사 (앱스토어 등록 불필요).
5. **서비스 소유권은 참가자에게** 있으며 공사는 개발 서비스를 운영하지 않는다. 이에 따라 **서비스 내 "한국관광공사", "KTO" 등 공사 지칭 단어·로고 사용 금지** (FAQ 명시). 데이터 출처 표기는 "TourAPI"로만.

## 사무국 질의 상태 (SPEC.md 7.1)

| # | 질의 | 상태 |
|---|---|---|
| ① | 좌표 무전송 구조(단말 내 처리 + 지역코드만 중계하는 프록시)의 위치정보법 비신고 해당 여부 | 발송 대기 |
| ② | 서버리스 프록시 경유 호출의 "실시간 호출" 인정 여부 | 발송 대기 |
| ③ | 전국형 제안서 → 부산 특화 구체화 허용 여부 + 지역특화 가점 인정 여부 | 발송 대기 |

- 회신 수신 시 이 표의 상태와 SPEC.md 5장(가점 행)·11장(리스크)을 갱신한다.
- 질의 ①②는 회신과 무관하게 개발 진행 가능 (좌표 무전송 구조가 어느 결론에서도 안전한 쪽).
- 질의 ③이 부정 회신이어도 부산 특화는 유지 (가점 없이 기획력·완성도 기여로 성립 — SPEC.md 11장).
