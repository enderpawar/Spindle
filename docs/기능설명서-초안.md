## 1. 서비스 개요

**Spindle은 휴대폰을 돌려(spin) 가리키는 방향의 부산 원도심·영도 관광지를 발견하는 관광 분산형 게임 탐색 PWA다.**

"다음에 어디 가지?"라는 여행 중 의사결정 피로를 **돌리기라는 우연성 게임**으로 바꾼다. 단순 랜덤 룰렛이 아니라, **대표 관광지 쏠림(오버투어리즘)을 골목·시장·근현대 문화유산으로 분산시키는 장치**로 설계했다.

- 핵심 메커니즘 한 줄: **"랜덤여행이라는 검증된 놀이 방식으로, 쏠린 관광객을 원도심으로 흘려보낸다."** 재미(수요)와 분산(정책 효과)이 같은 메커니즘 안에서 동시에 성립하는 것이 차별점이다.
- 서비스 유형: **웹(PWA)** — URL 접속만으로 심사 시연이 가능하고, 앱스토어 등록 리스크가 없다.
- 대상 지역: **부산 중구·동구·서구·영도구**(남포동·부산역·초량·영도 일대) — 지역특화.

## 2. 배경 — 왜 "관광 분산"인가

> ⚠ 아래 수치는 초안 작성 시점의 공개 자료 기반이며, 최종 제출본 전 원출처에서 재확인한다.

**① 쏠림 문제** — 부산 방문은 해운대·광안리·서면 소수 지점에 집중된다. 최근 조사에서 외국인 방문객의 절반 이상이 광안리에 몰렸고, 10~30대 젊은 방문자는 해변·번화가에 머문다. 반면 감천문화마을·자갈치 등 **원도심은 고령층 위주**로, 젊은 방문자를 원도심으로 보내는 통로가 없다.

**② 명분** — 서비스 대상지인 원도심·영도는 **인구 소멸위험 지역**이다(영도구는 광역시 기초단위 소멸위험지수 최저 수준). "관광 분산"이 수사가 아니라, 대상지 자체가 정책적 우선 지역이다.

**③ 순풍** — 정부·지자체의 관광 빅데이터 기반 도보투어 사업이 인구감소 대응으로 **동구·서구·영도구** 코스를 선정했고, 최근 원도심 외국인 방문 증가율이 급등했다. 분산은 이미 시작된 추세이며, Spindle은 이를 **우연이 아닌 구조**로 만드는 도구다.

**④ 수요** — "랜덤여행" 키워드 언급량이 최근 급증했고, 무계획·미스터리 여행이 2026 여행 트렌드에 등재됐다. Spindle의 스핀은 이 트렌드 심리를 정확히 겨냥한다.

## 3. 범위 구체화 경위

> 초기 전국형 기획을 현장 검증한 결과 추천 품질과 지역 관광분산 효과를 높이기 위해 부산 특화형으로 구체화했으며, 핵심 서비스 구조는 유지하였다.

최초 제안서는 전국 단위였으나 핵심 — **"현재 위치와 방향을 이용한 게임형 관광지 발견"** — 을 유지한 채 부산 특화로 범위를 구체화했다. 이는 **피벗이 아니라 구체화**이며, 제안서 기능별 반영 내역은 아래와 같다.

| 최초 제안서 기능 | 반영 | 비고 |
|---|---|---|
| 방향 가챠 (자이로+나침반, 8방위) | 유지 | §4.2 |
| 위치 기반 방향성 추천 | 유지 (부산 원도심·영도) | 좌표 무전송 구조 §7 |
| 거리 다이얼 | 스케일 조정 → 이동시간 다이얼 | §4.2 |
| 8방위 큐레이션 메시지 | 유지 | §4.2 |
| 공유 카드 자동 생성 | 이미지 카드로 유지 (회전 영상은 백로그) | §4.5 |
| 다녀온 곳 GPS 인증 컬렉션 | **부산 동네 도장깨기**로 변형 (방문 판정 단말 내, 서버엔 관광지 ID도 미전송) | §4.6 |
| 사전 캐싱 자체 DB | **실시간 `areaBasedList2` 호출**로 재설계 | 규정 준수 §6 |
| 인지도 낮은 관광지 가산점 | 유지 (분산 가중치) | §5.2 |

## 4. 핵심 기능

### 4.1 두 가지 모드

| 모드 | 출발점 | 센서 | 용도 |
|---|---|---|---|
| **여행 모드** | 부산역·남포동·영도 프리셋(지도 픽) | 센서 없이 `돌리기` 버튼 자동 스핀 | 여행 전 계획, **심사위원 시연 경로** |
| **현장 모드** | 단말 GPS(단말 내 처리) | DeviceOrientation — 휴대폰 회전에 맞춰 원판이 돌고 멈추면 방위각 확정 | 여행 중 즉흥 탐색 |

- **여행 모드는 센서·위치 권한 없이 완전 동작**하므로, 심사위원이 서울에서 URL만 열어도 전체 흐름이 성립한다.
- 두 모드 모두 좌표는 단말 밖으로 나가지 않는다(§7).

### 4.2 스핀 → 8방위

- 확정 방위각을 8방위(동·서·남·북·북동·북서·남동·남서, 각 45° 부채꼴)로 분류한다.
- 방위마다 고유 큐레이션 메시지로 연출한다 (예: 동 = "시작과 새벽의 방향, 떠오르는 해를 만나러").
- 화면 최종 각도와 알고리즘 입력 방위각은 항상 일치한다.

### 4.3 이동시간 다이얼 · 테마 덱

- **이동시간 다이얼**: 가볍게(도보 ~20분) / 반나절(~40분) / 하루 나들이(권역 전체). 기본값 반나절.
- **테마 덱**: 바다 / 골목·시장 / 근현대·역사 / 야간 / 먹거리 중 택1. 기본은 테마 없이 돌려 우연성을 보존한다.

### 4.4 결과 카드

- TourAPI 상세정보·대표 이미지, 보정 도보 예상 시간, 유형별 이용시간·휴무·체험·요금·주차 등 방문정보를 텍스트 중심 라벨–값 목록으로 제공한다. 인기도 티어는 추천 계산에만 사용하고 결과 UI에 배지로 표시하지 않는다.
- 이미지가 없으면 **방위 색상 배경 + 나침반 아이콘 폴백**으로 빈 화면을 만들지 않는다.
- 길찾기 딥링크(카카오맵·네이버지도), "다른 후보 보기", "다시 돌리기".
- **실패 처리**: 부채꼴 내 후보가 없으면 인접 방위로 자동 확장하고 사유를 명시한다("이 방향은 바다예요. 범위를 넓혔어요").

### 4.5 공유 카드

- 결과 화면을 인스타그램 스토리 비율(1080×1920) 이미지 카드로 자동 생성한다.
- Web Share API + 다운로드 폴백.

### 4.6 부산 동네 도장깨기 (2순위)

- 방문한 관광지를 **단말 localStorage에만** 기록하고 존(구)별 수집 현황을 보여준다.
- 방문 판정·저장이 전부 단말 내에서 일어나며, **좌표·식별자를 서버로 보내지 않는다.**

### 4.7 축제 특별 카드 (조건부, 2순위)

- 축제는 핵심 결과가 아닌 **조건부 특별 카드**다. 스핀 방위·기간이 맞을 때만 일반 결과 위에 등장한다.
- 행사 데이터가 없어도 일반 추천은 항상 동작한다 → 안정성 확보, 있으면 실시간성 증명.

### 4.8 방향 기반 여행 코스 (2순위)

- 단일 추천을 첫 장소로 고정하고 확정 방위·다이얼을 유지한 채 2~4개 장소를 연결한다.
- 코스 구성·순서·이동비용 계산은 **전부 단말 내**에서 수행하며 외부 경로 최적화 API를 쓰지 않는다.
- 다이얼별 목표: 가볍게 2곳 / 반나절 3곳 / 하루 4곳. 후보 부족 시 축소 코스, 2곳 미만이면 단일 추천 유지.

## 5. 추천 알고리즘 — "지역 특성이 알고리즘을 바꾼다"

후보 점수 = **방향 일치도 × 접근 가능성 × 운영 상태 × 분산 가중치**

| 요소 | 산정 방식 |
|---|---|
| 방향 일치도 | 8방위 부채꼴(45°) 포함 여부 + 방위 중심선에 가까울수록 가중 |
| 접근 가능성 | **존-교량 모델**(§5.1) — 바다·교량으로 "방향은 가깝지만 갈 수 없는 곳"을 보정거리로 제외 |
| 운영 상태 | 쉬는날·이용시간 파싱(불완전 데이터는 보수적 통과), 축제는 기간 필터 |
| 분산 가중치 | 인기도 티어 역가중(§5.2) — 덜 알려진 곳일수록 가중 ↑ |

### 5.1 존-교량 접근 가능성 모델 ⚠(존 다각형 확정 대기)

원도심·영도를 5개 존으로 구획한다. 바다·산·교량(영도대교·부산대교) 때문에 직선거리가 실제 이동가능성을 나타내지 못하므로, 존 간 이동은 교량·지하철 경유 보정거리로 재계산한다.

| 존 | 이름 | 포함 지역 |
|---|---|---|
| Z1 | 남포·광복 | 남포동·광복동·자갈치·국제시장 |
| Z2 | 부산역·초량 | 부산역·이바구길·차이나타운 |
| Z3 | 영도 서부 | 영도대교 남단·흰여울 |
| Z4 | 영도 동부 | 태종대 방면 |
| Z5 | 서구 | 송도·아미동 |

- **존 내 이동**: 직선거리 × 도보 우회계수 ≈1.3, 보정 도보시간 = 보정거리 ÷ 분속 67m(4km/h).
- **다이얼 상한**: 가볍게 20분 / 반나절 40분 초과 시 후보 제외. 바다 건너 직선 경로는 반드시 존 간 보정거리로 재계산.
- ⚠ 존 다각형 좌표·교량 경유점·존 간 보정 방식은 `docs/zones.md`에서 최종 확정 후 본 표에 지도로 싣는다.

### 5.2 인기도 티어 분산 가중치 (큐레이션 v0.1, 35곳)

| 티어 | 정의 | 분산 가중치 | 배지 |
|---|---|---|---|
| T1 | 대표 관광지(검색·SNS 최상위) | ×0.5 | — |
| T2 | 알려졌지만 쏠림 없는 곳 | ×1.0 | — |
| T3 | 숨은 명소(골목·시장·근현대 유산) | ×1.6 | "숨은 명소" |

- 큐레이션에 없는 관광지는 기본 T2. 대상 4개 구 대표 관광지 35곳을 존·테마·티어로 수동 큐레이션했다(`docs/curation.md`).
- **이 큐레이션 표에는 TourAPI 원본 응답(주소·좌표·소개·이미지 URL 등)을 저장하지 않는다.** 관광지 식별은 contentId로만 한다.

## 6. TourAPI 활용 내역

> 데이터는 **TourAPI(KorService2) 실시간 호출만** 사용한다. 로컬 DB·파일 적재 없이, 세션 시작 시점과 결과 시점에 실시간 호출한다.

| 용도 | 엔드포인트 | 비고 |
|---|---|---|
| 지역 POI 목록 | `areaBasedList2` | areaCode=6(부산) + 시군구코드(중·동·서·영도). **사용자 좌표 불필요** |
| 상세 정보 | `detailCommon2`, `detailIntro2` | 소개·이용시간·쉬는날 |
| 이미지 | `detailImage2` | 결과 카드 |
| 축제·행사 | `searchFestival2` | 지역코드+날짜. 좌표 불필요 |
| 지역코드 확인 | `areaCode2` | 시군구코드를 하드코딩하지 않고 이 API로 확인 |

- **`locationBasedList2`(좌표 기반)는 의도적으로 사용하지 않는다** — §7 좌표 무전송 원칙.
- 세션 시작 시 4개 구 목록을 실시간 호출해 **메모리/세션 캐시에만** 보관(영속 저장 없음). 상세·이미지·축제는 결과 시점에 추가 호출하여 개발·운영 기간 호출 이력이 자연히 쌓인다.

## 7. 아키텍처 — API 키 보호와 위치정보 보호의 동시 해결

**원칙: 사용자 좌표·방위각은 어떤 서버로도 전송하지 않는다. 프록시는 인증키만 보호한다.**

```
[PWA 클라이언트]                       [프록시 (서버리스 엣지)]            [TourAPI]
 GPS·방위각 → 단말 내 계산만            인증키 주입 + 요청 중계만
 "부산 중구 POI 목록 줘" ──────────▶   (좌표 수신 자체가 없음) ────────▶  areaBasedList2
 방향 매칭·거리·필터 = 전부 단말        접근 로그·파라미터 미저장
```

- 요청 파라미터가 **지역코드뿐**이므로 프록시가 개인위치정보를 수신·처리하지 않는다. "서버 프록시 = 좌표 전송"이라는 충돌 구도 자체를 제거했다.
- 인증키는 프록시 환경변수로만 보관 → 클라이언트·저장소·커밋에 노출 없음.
- 프록시(Cloudflare Workers)는 **허용 엔드포인트·파라미터 화이트리스트**로 좌표 파라미터가 담긴 요청을 400으로 거부한다. 금지 패턴 스캐너가 커밋·편집 단계에서 `locationBasedList2`·클라이언트 내 키·좌표 파라미터를 기계적으로 차단한다.

---

## 8. 완성도·안정성

- **두 모드 폴백**: 센서 미지원·권한 거부 시 여행 모드로 자연 유도. 데스크톱에서 완전 동작.
- **실패 처리**: 후보 없는 방위에서 인접 확장 + 사유 문구.
- **에러/로딩**: 모든 TourAPI 호출 지점에 로딩 상태, 모든 실패 경로에 에러 UI + 재시도(빈 화면 금지).
- **PWA**: manifest(standalone·portrait·마스커블 아이콘), 서비스워커는 **정적 자산만 프리캐시**하고 API 경로는 network-only. 오프라인 시 셸 + "네트워크 필요" 안내(추천은 동작하지 않음 → 실시간 호출 규정 준수 신호).
- 품질 게이트: 타입체크·린트·유닛테스트·금지 패턴 스캔을 pre-commit에서 강제.

---

## 9. 발전성·확장성

- **타 지역 템플릿화**: 존 모델·큐레이션이 코드가 아닌 **데이터**(`zones.md`·`curation.md`)다. 원도심형 도시(대전·광주 동구 등)에 데이터만 교체해 복제할 수 있다.
- **RTO 협업 모델**: 익명 집계(방위·티어별 추천/선택 카운트, 좌표·식별자 미저장)로 분산 유도 효과를 지표화하는 로드맵. *(규정 검토 후 도입 예정)*
- **제안서 발전방향 승계**: 계절·시간대 가중치, 동반자 유형 필터, 반려동물 모드, 다국어(인바운드), 공유 카드 회전 영상 캡처.
