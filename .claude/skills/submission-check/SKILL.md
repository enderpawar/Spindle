# 제출 전 점검 절차

마감: **2026-09-21(월) 16:00 정각**. 목표: 마감 최소 1일 전 제출 완료.
아래 항목을 실제 코드·문서·배포 상태에 대해 검사하고, 통과/미통과와 근거를 표로 보고한다.

## 1. 제출물 3종

- [ ] **서비스 URL**: 배포 URL이 고정되어 있고, 시크릿 창 + 데스크톱 브라우저(센서 없음)에서 여행 모드로 전체 흐름(스핀→추천→결과→공유 카드)이 완주되는가
- [ ] **기능설명서**: 아래 필수 포함 항목 확인
  - 존-교량 모델 지도와 가중치 표 (docs/zones.md, docs/curation.md 기반)
  - TourAPI 활용 내역 (엔드포인트별 용도, 실시간 호출 구조도)
  - 범위 구체화 경위 문장: "초기 전국형 기획을 현장 검증한 결과 추천 품질과 지역 관광분산 효과를 높이기 위해 부산 특화형으로 구체화했으며, 핵심 서비스 구조는 유지하였다."
  - 비로그인 서비스임을 명시 (테스트 계정 불필요)
- [ ] **TourAPI 운영계정 정보**: 운영계정 키로 개발 기간 중 호출 이력이 쌓여 있는가 (개발용 별도 키 사용 흔적 없는지)

## 2. 규정 준수 (감점/제외 사유 검사)

- [ ] 코드 전역에서 `locationBasedList2` 검색 → 0건
- [ ] 코드 전역에서 API 키 문자열·serviceKey 하드코딩 검색 → 0건 (프록시 환경변수만)
- [ ] localStorage/IndexedDB/정적 파일에 TourAPI 응답 적재 코드 → 0건
- [ ] service worker가 API 경로를 캐싱하지 않음 (network-only 확인)
- [ ] 네트워크 요청 페이로드에 좌표·방위각 미포함 (프록시 라우트 화이트리스트 확인)
- [ ] 코드·UI 문구·메타태그·공유 카드·기능설명서에서 "한국관광공사", "KTO", 공사 로고 검색 → 0건 (데이터 출처는 "TourAPI"로만 표기)

## 3. 완성도 (심사 30점 직결)

- [ ] iOS Safari 실기기: 현장 모드 전체 흐름 + 권한 거부 시 폴백
- [ ] Android Chrome 실기기: 동일
- [ ] 실패 처리: 후보 없는 방위에서 인접 확장 + 사유 문구 표시
- [ ] TourAPI 장애·타임아웃 시 에러 UI (빈 화면 금지)
- [ ] PWA: manifest·설치 배너·오프라인 셸 (정적 자산만) 동작
- [ ] 로딩 상태가 모든 API 호출 지점에 존재

## 4. 최종 확인

- [ ] SPEC.md와 실제 구현의 불일치 항목 목록화 → 기능설명서에 구현 기준으로 서술
- [ ] 사무국 질의 회신(가점 인정 여부) 반영 — 미회신이면 기능설명서에서 가점을 전제하지 않는 서술 유지
- [ ] 제출 완료 후 URL 최종 재접속 확인
