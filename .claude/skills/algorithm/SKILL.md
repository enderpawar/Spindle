# 추천 알고리즘 구현 규약 (SPEC.md 4장)

## 점수 산식

```
후보 점수 = 방향 일치도 × 접근 가능성 × 운영 상태 × 분산 가중치
```

모든 계산은 **단말 내에서만** 수행한다 (좌표 무전송). 곱셈 구조이므로 어느 한 요소가 0이면 후보 제외.

## 1. 방향 일치도

- 8방위 부채꼴(45°) 판정과 방위각 수학은 `sensors` 스킬 규약을 따른다 (원형 경계 처리 포함).
- 부채꼴 밖 = 0 (제외). 부채꼴 안 = `0.5 + 0.5 × (1 - |Δ중심선| / 22.5)` — 경계에서도 0.5는 보장해 부채꼴 내 후보가 지나치게 좁혀지지 않게 한다.
- POI 방위각은 출발점 → POI 좌표(TourAPI 응답 mapx/mapy)의 방위. 세션 내 재계산 비용이 낮으므로 사전 계산·영속화하지 않는다.

## 2. 접근 가능성 — 존-교량 모델

- 존 다각형·존 간 보정거리·우회계수는 **`docs/zones.md`가 단일 진실 원천**. 코드에 수치를 중복 하드코딩하지 말고 해당 데이터 구조를 모듈로 분리해 zones.md와 항상 동기화한다 (수치 변경 시 zones.md도 갱신).
- 같은 존: 직선거리 × 1.3 / 분속 67m → 보정 도보시간.
- 다른 존: 존 간 연결 정의(교량 경유 경로거리 또는 대중교통 고정 시간)를 적용. 연결 정의가 없는 존 쌍은 "하루(무제한)"에서만 후보 허용.
- 이동시간 다이얼: 3단 고정이 아니라 **20분~하루를 자연스러운 눈금(20·30·40·60·90·120·180·240분·하루)의 분 예산**으로 조정한다 (`mock/pois.ts DIAL_STEPS`). 예산 초과 = 0 (제외), 하루 = Infinity(권역 내 무제한). 경계값 정각은 포함.

## 3. 운영 상태

- `detailIntro2`의 쉬는날·이용시간 파싱 실패 시 **보수적으로 통과(1.0)** — 데이터 불완전을 이유로 후보를 죽이지 않는다. 파싱 성공 & 휴무 확정 시 0.
- 축제(searchFestival2)는 점수 요소가 아니라 **조건부 특별 카드**: 스핀 방위 부채꼴 안 + 오늘이 기간 내일 때만 일반 결과 위에 별도 카드. 축제 데이터가 없어도 일반 추천은 항상 동작해야 한다.

## 4. 분산 가중치

- `docs/curation.md`의 티어 표 기준: T1 ×0.5 / T2 ×1.0 / T3 ×1.6. 표에 없는 contentId는 T2.
- T3 당첨 시 결과 카드에 "숨은 명소" 배지.

## 후보 선정·실패 처리

- 점수 상위 **3개를 점수 비례 가중 랜덤**으로 1곳 선정. "다른 후보 보기"는 나머지 2개 순환, "다시 돌리기"는 전체 재스핀.
- 부채꼴 내 후보 0개 → **인접 방위(±45°)로 확장**하고 사유 문구를 반드시 표시 ("이 방향은 바다예요. 범위를 넓혔어요"). 확장 후에도 0개면 반대편 제외 전방위로 확장.
- 직전 결과와 동일 POI 연속 당첨 방지: 같은 세션에서 직전 1개는 후보에서 임시 제외.

## 테스트

- 랜덤 선정은 RNG를 주입 가능하게 만들어 시드 고정 테스트를 작성한다.
- 필수 테스트 케이스: 방위 경계(0°/360° wrap), 바다 방향(남포에서 남쪽) 확장 동작, 다이얼별 제외 경계값, 존 간 교량 보정이 직선거리보다 커지는 케이스.
