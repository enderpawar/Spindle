> 휴대폰을 돌려, 가리키는 방향의 관광지를 발견하는 **관광 분산형 게임 탐색 PWA**

## 1. 서비스 정의

Spindle은 "다음에 어디 가지?"라는 여행 중 의사결정 피로를 **돌리기(spin)** 라는 우연성 게임으로 바꾸는 서비스다.
단, 단순 랜덤 룰렛이 아니라 **대표 관광지 쏠림(오버투어리즘)을 골목·시장·근현대 문화유산으로 분산시키는 게임형 탐색 장치**로 정의한다.

- 심사 서사: "재미있는 룰렛"이 아니라 "관광객 분산 문제를 게임화로 푸는 서비스"
  - 1차 심사 기획력(트렌드·독창성 30점), 발전성(20점) / 최종 심사 시의적절성(30점) 대응
- 서비스 유형: **웹(PWA)** — URL 제출만으로 심사 가능, 앱스토어 등록 리스크 회피
- 지역특화: **부산 원도심·영도권** (가점 +2 — 전환 후 인정 여부 사무국 확인 중, 부산 RTO 특별상 후보권)

### 1.1 최초 제안서(전국형)와의 관계

최초 제안서는 전국 단위였으나, 핵심 — **"현재 위치와 방향을 이용한 게임형 관광지 발견"** — 을 유지한 채 부산 특화로 범위를 **구체화**한다. 1차 심사에 초기 제안서와의 일치도 배점은 없으며, 심사는 최종 제출물(완성 서비스 + 기능설명서) 기준이다.

- 사무국 서면 질의로 전환 허용 및 지역특화 가점 인정 여부를 확인한다 (7.1-③)
- 기능설명서에 변경 경위를 숨기지 않고 한 문장으로 명시한다:
  > "초기 전국형 기획을 현장 검증한 결과 추천 품질과 지역 관광분산 효과를 높이기 위해 부산 특화형으로 구체화했으며, 핵심 서비스 구조는 유지하였다."
- 제안서 기능별 반영 내역은 3.3 표 참조 — 이 표가 "피벗이 아닌 구체화"의 근거 자료가 된다.

## 2. 대상 지역 — 부산 원도심·영도

전국 서비스의 지역 필터가 아니라, **지역의 물리적 특성이 추천 알고리즘 자체를 바꾸는** 설계를 채택한다.

- 범위: 부산 중구·동구·서구·영도구 (남포동, 부산역, 초량, 영도 일대)
  - 지역특화 인정 기준(기초자치단체 단위, 복수 기초단체 묶음 가능) 충족
- 선정 근거:
  - POI 밀도가 높고 도보·대중교통 이동권 → "돌려서 나온 곳에 실제로 간다"가 성립
  - 해운대·광안리로의 쏠림 대비 원도심의 골목·시장·근현대 콘텐츠 발견 서사가 선명
  - 부산관광공사(RTO) 존재 → 특별상 심사 대상 지역
- 지역 특성 = 알고리즘 제약: 바다·산·교량(영도대교, 부산대교)으로 인해
  "방향상 가깝지만 실제로는 갈 수 없는 장소"가 빈발 → 3장의 접근 가능성 모델로 해결

## 3. 사용자 흐름

### 3.1 두 가지 모드

| 모드 | 출발점 | 센서 | 용도 |
|---|---|---|---|
| **현장 모드** | 단말 GPS (단말 내 처리) | 방향: DeviceOrientation. 휴대폰의 실제 회전 속도에 맞춰 원판이 움직이고, 정지 후 안정된 방위각을 확정 | 여행 중 즉흥 탐색 |
| **여행 모드** | 부산역 / 숙소 / 지도에서 찍은 지점 | 센서 없이 `돌리기` 버튼으로 자동 스핀 | 여행 전 계획, **심사위원 시연 경로** |

- 여행 모드가 있어야 심사위원이 서울에서 URL만 열어도 서비스가 완전하게 동작한다.
- 현장 모드에서만 센서·GPS를 사용하며, 두 모드 모두 좌표는 단말 밖으로 나가지 않는다 (7장).
- 모바일 UI는 280~480px 폭을 지원한다. 홈의 테마·추천 카드는 고정폭 가로 목록 대신 폭에 따라 2~3열로 재배치하며, 헤더·진행률·긴 문구가 화면 밖으로 잘리지 않아야 한다.
- 첫 방문 온보딩 이후 설정의 [사용법 다시 보기]는 별도 슬라이드가 아니라 실제 홈 UI의 출발점·스핀·테마·도장 요소를 강조하는 단계형 인앱 코치마크로 제공한다.
- 화면 이동은 320~460ms의 겹침 전환을 쓰되 좌우 슬라이드 없이 깊이(z축) 큐로 위계를 표현한다: 탭 간은 잔잔한 디졸브, 하위 진입은 zoom-in, 상위 복귀는 zoom-out(모두 이전 화면 고정 + 새 화면 페이드인)이며 미지원 브라우저는 동일한 진입 모션으로 폴백한다. 버튼은 빠른 누름(70ms)과 또렷한 스프링 복원(1.0을 약 6% 지나쳤다 정착)을 조합한 역할별 피드백을 제공하고, 리빌·결과·도장·탭 선택 등 히어로 순간은 살짝 지나쳤다 정착하는 팝으로 강조하며, 시스템의 모션 최소화 설정에서는 이동·확대·팝 효과를 제거한다.

### 3.2 추천 흐름

1. **(선택) 이동시간 다이얼** — 3단 고정이 아니라 20분~하루(권역 내 무제한)를 자연스러운 눈금(20·30·40·60·90·120·180·240분·하루)으로 좌우 조정한다. 값은 분 단위 이동시간 예산으로 엔진에 전달되며, 연결 미정의 존 쌍은 "하루"에서만 허용. 기본값 40분. [제안서 기능3의 부산 스케일 버전]
2. **(선택) 테마 덱** — 바다 / 골목·시장 / 근현대·역사 / 야간 / 먹거리 중 택1. 기본은 테마 없이 바로 돌리기 (우연성 보존)
3. **스핀 → 8방위 분류** — 현장 모드는 휴대폰의 실제 회전 속도에 맞춰 원판을 움직이고, 기기 회전이 멈춘 뒤 안정된 실제 방위각을 확정한다. 여행 모드는 센서 없는 자동 스핀으로 방위각을 정한다. 확정 각도를 8방위(동·서·남·북·북동·북서·남동·남서, 각 45° 부채꼴)로 분류하고, 방위별 고유 큐레이션 메시지와 함께 연출한다. 화면 최종 각도와 알고리즘 입력 방위각은 항상 같아야 한다. (동: "시작과 새벽의 방향, 떠오르는 해를 만나러") [제안서 기능1·4]
4. **결과: 가중 랜덤** — 해당 방위 부채꼴 안에서 접근 가능한 후보 중 상위 3개를 가중 랜덤으로 1곳 제시. "다시 돌리기" / "다른 후보 보기" 제공
5. **실패 처리** — 부채꼴 내 후보 없으면 인접 방위로 자동 확장하고, 확장 사유를 화면에 명시 ("이 방향은 바다예요. 범위를 넓혔어요")
6. **결과 상세** — TourAPI 상세정보·이미지, 도보 예상 시간, 길찾기 딥링크(카카오맵/네이버지도). 기본 결과 카드부터 주소·이동·운영·휴무 표를 설명보다 먼저 노출하고, `overview` 앞부분은 3~4줄로 요약하며 전체 원문은 [자세히 보기]에서 제공
7. **공유 카드** — 결과 화면을 인스타그램 스토리 비율의 이미지 카드로 자동 생성 [제안서 기능5, 영상 캡처는 백로그]
8. **(2순위) 방향 기반 여행 코스** — 단일 추천 결과에서 "이 방향으로 코스 짜기"를 선택하면 확정 방위와 이동시간 다이얼을 유지한 채 2~4개 장소를 연결한다. 코스 구성·순서·이동비용 계산은 전부 단말 내에서 수행하며 상세 규칙은 `docs/course.md`를 따른다. 길찾기 시 새 도장을 얻으면 지도 선택 시트 내부에서 인라인으로 안내한다.

## 4. 추천 알고리즘

후보 점수 = **방향 일치도 × 접근 가능성 × 운영 상태 × 분산 가중치**

| 요소 | 산정 방식 | 데이터 |
|---|---|---|
| 방향 일치도 | 8방위 부채꼴(45°) 내 포함 여부 + 방위 중심선에 가까울수록 가중 | 단말 계산 |
| 접근 가능성 | **존(zone)-교량 모델**: 원도심·영도를 4~6개 존으로 수동 구획, 같은 존은 직선거리×도보 우회계수(≈1.3), 존 간 이동은 교량·지하철 경유 보정거리. 보정 도보시간 상한(예: 40분) 초과 시 제외 | 수작업 존 정의 (지역특화의 핵심 근거) |
| 운영 상태 | 쉬는날·이용시간 파싱(불완전 데이터는 보수적으로 통과), 축제는 기간 필터 | TourAPI detailIntro2 / searchFestival2 |
| 분산 가중치 | 대표 관광지 20~30곳을 인기도 티어로 수동 큐레이션, 티어 역가중(덜 알려진 곳일수록 가중 ↑). 결과 화면에 "숨은 명소" 배지 | 수작업 큐레이션 |

- 존-교량 모델과 인기도 큐레이션이 곧 **"지역 특성 때문에 알고리즘이 달라진다"** 는 심사 논거의 실체다. 기능설명서에 존 지도와 가중치 표를 그대로 싣는다.

### 4.1 특별 카드 — 축제·행사

- 축제는 핵심 결과가 아닌 **조건부 특별 카드**: 스핀 방향·기간이 맞을 때만 일반 결과 위에 "오늘 열리는 행사" 카드로 등장
- 행사 데이터가 없어도 일반 추천은 항상 동작 → 안정성 확보, 있으면 실시간성 증명

### 4.2 방향 기반 여행 코스 (2순위)

- 단일 추천 결과를 첫 장소로 고정하고, 확정된 기준 부채꼴에서 후속 장소를 선정한다. 후보가 부족할 때만 양쪽 인접 방위로 확장하고 사유를 표시한다.
- 이동시간 예산별 목표 장소 수는 20분 이하 2곳 / 1시간 이하 3곳 / 그 이상·하루 4곳이다. 총 이동비용은 `docs/zones.md`의 존-교량 모델로 계산하며 다이얼의 분 예산을 상한으로 적용한다 (하루는 권역 내 무제한).
- 목표 수를 채우지 못해도 2곳 이상이면 축소 코스를 제공하고, 2곳 미만이면 기존 단일 추천 결과를 유지한다.
- 외부 경로 최적화 API 없이 세션 메모리의 실시간 POI 후보만 사용한다. 사용자 좌표·방위각은 서버로 전송하지 않고 코스 및 API 응답을 영속 저장하지 않는다.
- 후보 자격·순서·동률·실패 처리의 단일 진실 원천은 `docs/course.md`다.

## 6. 데이터 — TourAPI (KorService2)

> **필수 제약**: 데이터는 TourAPI 실시간 호출만 인정. 로컬 DB 적재·파일 데이터 사용 시 감점/제외.
> 운영계정 인증키 기준으로 **개발 기간 중 실제 호출 이력을 검증**하므로, 개발 첫 주부터 운영계정 키로 호출한다.

| 용도 | 엔드포인트 | 비고 |
|---|---|---|
| 지역 POI 목록 | `areaBasedList2` | areaCode=6(부산) + 시군구코드(중·동·서·영도구). **사용자 좌표 불필요** |
| 상세 정보 | `detailCommon2`, `detailIntro2` | 소개·이용시간·쉬는날 |
| 이미지 | `detailImage2` | 결과 카드 |
| 축제·행사 | `searchFestival2` | 지역코드+날짜 기반. 좌표 불필요 |
| 지역코드 조회 | `areaCode2` | 시군구코드는 하드코딩하지 말고 이 API로 확인 |

- `locationBasedList2`(좌표 기반 조회)는 **의도적으로 사용하지 않는다** — 7장의 좌표 무전송 원칙 때문.
- 세션 시작 시 대상 4개 구의 POI 목록을 실시간 호출로 가져와 메모리/세션 캐시에만 보관(영속 저장 없음). 상세·이미지·축제는 결과 시점에 추가 호출 → 호출 이력도 자연스럽게 쌓인다.

## 7. 아키텍처 — API 키 보호와 위치정보법의 동시 해결

**원칙: 사용자 좌표는 어떤 서버로도 전송하지 않는다. 프록시는 키만 보호한다.**

```
[PWA 클라이언트]                         [프록시 (서버리스 엣지)]              [TourAPI]
 GPS·방위각 → 단말 내 계산만              인증키 주입, 요청 중계만
 "부산 중구 POI 목록 줘" ────────────▶   (좌표 수신 자체가 없음)  ────────▶  areaBasedList2
 방향 매칭·거리·필터 = 전부 단말          로그에 IP·파라미터 미저장
```

- 요청 파라미터가 지역코드뿐이므로 **프록시가 개인위치정보를 수신·처리하지 않는다.** "서버 프록시 = 좌표 전송"이라는 충돌 구도 자체를 제거.
- 인증키는 프록시 환경변수로만 보관 → 클라이언트 노출 없음. (키가 노출되면 타인 호출이 우리 호출 이력에 섞여 검증 신뢰도 문제도 생김)
- 프록시는 Cloudflare Workers 또는 Vercel Edge Functions, 접근 로그 비활성화.
- IP 주소는 통신상 불가피하게 전달되나 좌표·식별자와 결합하지 않으므로 리스크 최소화. 단, 아래 질의로 공식 확인을 병행한다.



## 8. 기술 스택 (제안)

- 프론트: React(Vite) + TypeScript, PWA(manifest + service worker, 오프라인 셸)
- 센서: DeviceOrientation(iOS 13+ 권한 프롬프트 필수 처리, `webkitCompassHeading` 폴백), Geolocation
- 프록시: Cloudflare Workers (무료 티어, 로그 미저장 설정)
- 지도/길찾기: 결과 화면에서 카카오맵·네이버지도 딥링크 (자체 지도 SDK는 여행 모드 출발점 선택에만 최소 사용)
- 배포: Cloudflare Pages 또는 Vercel — 심사 제출용 고정 URL

## 10. 우선순위 / 비범위

**MVP (1순위)**: 스핀 → 8방위 분류 → 이동시간 다이얼 → 추천(가중 랜덤·실패 처리) → 결과 카드 → 공유 이미지 카드, 여행 모드
**2순위**: 방향 기반 여행 코스, 테마 덱, 축제 특별 카드, 부산 동네 도장깨기(방문 컬렉션). Phase 7에서는 이 순서대로 검토하고 시간에 따라 뒤 항목을 생략한다.

**백로그 / 비범위**:
- 공유 카드의 회전 영상 캡처 (MediaRecorder) — 이미지 카드 안정화 후 검토
- 방언 카피 등 표면적 로컬라이징 (지역성은 알고리즘·큐레이션으로 증명)
- 로그인·계정 — 심사 편의상 비로그인 유지 (테스트 계정 제출 불필요)
- 계절·시간대 가중치, 동반자 유형 필터, 반려동물 모드, 다국어(인바운드) — 제안서 "발전방향" 그대로 기능설명서 발전성 항목에 기술
- 리뷰·커뮤니티, 다지역 확장 — 발전성 로드맵으로만 기술
- 생성형 AI 기능 — 감점 없음 확인됨, 완성도 우선이라 필요 시 결과 카드 한 줄 소개 생성 정도만 검토
